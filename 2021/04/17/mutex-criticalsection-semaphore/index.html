mutex, criticalsection, semaphore

Posted on 2021-04-17 | In fundamental

介绍用于mutal exclusion的几个概念，并介绍几种implementation。最后列一下C++11多线程支持库，并举一个线程池的例子。 ## Comcepts ### mutex 就是lock，用于mutual exclusion。mutex一般是系统调用。 ps: 语言层面的eastl::mutex使用系统的critical section API，API中使用系统内核的mutex。 ### critical section 临界区一般由mutex控制，在进入代码段前lock并在执行完后unlock。 ### semaphore 信号量简单的理解就是有counter的mutex，同样也是系统调用。是一个共享的int变量s，值非负。只能通过wait(s)和signal(s)访问。 - P wait(s) decrements s if s > 0; if not, the process is suspended. - V signal(s) increments s. wait(s) and signal(s) are done atomically.

生产者消费者问题

生产者生产对象给消费者，使用一个最大容量为N的队列通信。 - 队列为空，消费者必须等待生产者生产 - 队列为满，生产者必须等待消费者消费 用三个信号量解决这个问题： - emptyCount ：队列中的空余容量 - fullCount : 队列中已用容量 - useQueue : 控制队列的访问（可用mutex）

初始值emptyCount = N，fullCount = 0，useQueue = 1.

emptyCount可能小于实际空余容量，fullCount可能小于实际使用容量。 emptyCount + fullCount <= N永远成立。 ### non-blocking, lock-free, wait-free - non-blocking : 任何线程的suspension不会导致其他线程的failure或者suspension。 - lock-free : guaranteed system-wide progress(no starvation) - wait-free : guaranteed per-thread progress

Implementation

test-and-set

test-and-set-lock(TSL)可以由双端口存储器硬件实现，也可以由指令集在软件层面实现。 test-and-set指令：对一个内存位置写入1(set)，并返回该位置原来的值。这一整套使用单指令(原子执行)。 while(test_and_set(&lock) == 1)可作为简单的lock()函数。当返回值为0时，则已获得锁，退出循环，否则继续忙等(spin lock)。

### test and test-and-set 在系统编程常用，应在高层的并发编程中避免，容易引发double-checked locking test_and_set可能会导致忙锁的资源竞争(总线锁，频繁的原子访问内存导致的cache失效)，为了降低overhead使用test and test-and-set。 ### fetch-and-add FAA是原子操作，原子地在一个内存位置并加上一个值，并返回旧值。可以用于实现信号量和互斥锁。C++11有std::atomic<T>::fetch_add。 ### load-link/store-conditional (LL/SC) 也叫load-reserved/store-conditional(LR/SC)。LL和SC是多线程同步的一对指令。load-link返回一个内存位置的现有值，store-conditional只在LL之后该位置没有修改的情况下写入新值。LL/SC是指令集实现的。 ### compare-and-swap CAS指令，只在一个内存位置与指定值相同的情况下写入新值（原子执行），返回比较的结果（bool）。 ## C++11 multithreading ### future 提供访问异步操作结果的机制。not copyable - 使用(由async,packaged_task,promise创建的)异步操作可以提供一个future - 异步操作的创建者可以valid()，wait()或者get()，如果异步操作还没有提供值就会阻塞 - 当异步操作在准备好返回值给创建者的时候，可以使用与创建者的future关联的promise::set_value

async

异步运行一个可调用对象f，返回future。launch可以指定是此线程lazy或者新线程异步。

### packaged_task 提供可以被异步invoke的可调用对象Callable(function, lambda expression, bind expression, or another function object)的包装。返回future(值或异常)packaged_task.get_future()。not copyable ### promise promise 是promise-future通信的推进端，设置promise状态与获取future状态同步。not copyable ### condition_variable 同步原语，用于阻塞一个或多个线程，直到另一个线程修改共享变量(the condition)，并notify(condition_variable)。 not copyable, not movable #### 线程修改变量 - 获得mutex(一般通过lock_guard) - 在获得锁期间做修改 - notify_one()或notify_all()(无需在获得锁期间)

即使共享变量是atomic的，也必须在获得mutex的情况下修改，来保证等待线程能获得正确结果 #### 线程等待变量 - 获得unique_lock<mutex> - 检查共享变量(the condition)，防止已经被修改并且notify - wait(),wait_for()或wait_until()，原子地释放锁，并暂停线程执行。 - 当condition_variable notify后，原子地获得mutex

condition_variable只与unique_lock<mutex>配合(效率最大化)。condition_variable_any可与其他锁对象配合。

## LLVM ThreadPool llvm里的一个简单的线程池，异步并行地执行一定数量的线程。async用来向池中添加任务。使用了两个condition_variable，一个用来控制任务序列的访问，一个用来标记任务队列全部完成。 具体的实现区分了是否使用多线程，如果开启了多线程，则用vector里的线程不断的找任务队列里的任务执行。 - 加入任务时，QueueCondition.notify_one()唤醒一个正在等待任务的线程 - 工作线程内，在任务队列不空的时候，不断取任务执行。注意一下工作线程执行完任务后，都要减少ActiveThreads，并判断自己是不是最后一个Active的线程以及任务队列是否已空，如果是，那么发出CompletionCondition.notify_all()提醒线程池的ThreadPool::wait() - wait()用于等待任务队列清空(但并不销毁线程池)，清空后的线程池里工作线程会阻塞在 QueueCondition.wait(LockGuard, [&] { return !EnableFlag || !Tasks.empty(); }); - 线程池析构时，先将EnableFlag置false，然后使用QueueCondition.notify_all()唤醒所有还在等待任务的线程，接着等待工作线程把任务队列清空后销毁线程(join())。

如果是单线程，就只用launch::deferred的async在本线程里lazy evaluation。为了和多线程统一，就用package_task包装future.get()后加入任务队列。
