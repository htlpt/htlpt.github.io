[6.828] xv6 Virtual Memory

Posted on 2020-10-03 | In OS

跟着6.828把xv6看到了第3章，总结一下一到三章以及LEC 1-7。 ## Prologue 当PC启动时，BIOS初始化所有硬件设备后，将bootloader载入内存并开始执行。bootloader对寄存器做初始化，开启A20，将GDT地址输入到GDTR，使能CR0_PE，将CPU从16位模式转换到32位保护模式。然后调用bootmain。以下的bootmain将kernel的ELF文件载入内存，并找到entry开始执行。

kernel的entry做的第一件事是开启硬件的分页功能entry然后将事先写好的第一个页表entrypgdir的地址输入到CR4。最后打开CR0_PG，CR0_WP，自此开始，所有代码中的地址在执行被硬件执行时，都会被认为是虚拟地址，在翻译成物理地址后执行。开启分页后，将控制移交main中，准备开启第一个进程。 ## Virtual Memory 每一个进程都有一个页表，页表中包含着对所有虚拟内存(进程的地址空间2^32)的映射。xv6的页大小为4096 Bytes(2^12)，因此采用20 bit进行页的索引。x86采用二级页表，每级由10 bit索引，每个页表的大小为页的大小(10244=4096 Bytes)，页表中的表项包含20 bit的下级索引和10 bit的flag。一个虚拟地址使用前20 bit通过二级页表得到页的物理地址，使用虚拟地址最后12 bit作为偏移。 其中，每个进程的页表中的kernel部分的映射都相同，而用户空间0~KERNBASE在每个进程的页表中可以有自己的映射。运行时的内存分配只允许使用 kernel data end~PHYSTOP的不到224MB的空间*。 kernel空间的mapping如下图所示，可见PHYSTOP到DEVSPACE，即0xE000000~0xFE000000对应的内存不会被使用。

以下是第一个一级页表entrypgdir，其中只有两个表项，首先将用户空间的虚拟地址0~0x400000和kernel空间的0x80000000~0x80400000映射到物理地址的0~0x400000。注意这里设置PTE_PS，映射的是super page(大小为4M Bytes)。因为这样一个页表，使得entry的指令在分页开启后仍然在低物理地址执行，也就是bootloader载入的位置(低虚拟地址映射到低物理地址)。跳转main后，用户空间映射entrypgdir[0]将被移除，kernel空间映射entrypgdir[512]将继续被使用。

First Process

接之前的bootmain,控制转移到main。

### Init Memory Mapping xv6使用一个kmem.freelist链表来记录可用的内存页，实现物理内存的管理。这里有一个引导的问题，物理内存需要被页表映射后才能被allocator的freelist初始化(freelist记录的都是虚拟地址)，但是建立页表也需要从freelist里分配内存。

xv6的解决办法是使用单独的allocator，即kinit1利用之前entrypgdir[512]的表项(一个4M的super page，映射0~0x400000)，来把kernel data end~0x80400000的内存直接添加到freelist。这样allocator有了内存，可以来为整个操作系统真正的一级页表开辟空间。

接着kmalloc()为整个操作系统创建一个一级页表kpgdir，并由setupkvm填好kernel部分的映射，然后切换到这个新的一级页表。kernel部分的映射主要包括kmap中的四段。 kinit1之后又经历了多部初始化，kinit2初始化了0x80400000~0x80E000000的内存并将锁开启，将所有可用内存都交给allocator初始化，kpgdir包含了所有的可用内存映射。之后对kalloc()的调用，都将使用kpgdir。值得多提一句，在entrypgdir没有被替换时，这些bootstrap的代码都在KERNBASE+4M空间内执行，所有的映射建立都只是对虚拟/物理地址数值的处理（只是填表），所以并不会出现PAGE FAULT。

Prepare Content

之后的userinit初始化第一个进程，其中使用了一些技巧让这个进程就像是普通的中断或陷入的进程。

首先allocproc遍历一个全局的进程描述符数组，获得一个UNUSED进程描述符struct proc。接着对进行描述符进行初始化，主要值得注意的是kernel stack。 kernel stack分布如下，希望达到的效果是该进程在进入kernel时先执行forkret，并且使用context内的寄存器，这一步可以将p->context->eip设置为forkret实现。下一步要让forkret执行完后执行trapret然后回到用户空间。forkret执行完毕后会回到栈顶保存的地址(应该保存的eip)，将栈顶保存为是trapret就可以达到目的。

之后setupkvm设置页表的kernel部分映射。inituvm将init程序载入到虚拟地址0x00000000并建立映射。之后设置trapframe，伪造一个进入中断/陷入的上下文，设置p->tf->cs = (SEG_UCODE << 3) | DPL_USER，p->tf->es = p->tf->ss = p->tf->ds = (SEG_UDATA << 3) | DPL_USER，使得回到用户空间时是用户态。设置p->tf->eip = 0，使得现场还原后直接开始执行init，最后将进程状态设为RUNNABLE。

Running the First Process

mpmain中调用了scheduler。scheduler找到一个RUNNABLE的进程(唯一一个initproc)，switchuvm切换页表和TSS。

之后swtch直接替换context，swtch函数中ret使用的eip就指向之前设置的forkret。当forkret返回的ret使用的eip就是trapret(将栈顶的地址作为eip)，trapret还原“现场”，将eip置为0，esp置为4096，回到用户空间和用户态，开始执行initcode.s。 ## First System Call : exec 回到用户空间0x00000000后开始执行initcode.S，首先压栈$argv，$init，0，将系统调用类型（SYS_exec）移入eax，之后用int指令调用T_SYSCALL。 执行时esp为4096，用户内存布局如下:
